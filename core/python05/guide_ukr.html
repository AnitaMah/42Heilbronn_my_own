🌆 Реальна Аналогія: Код Нексус
Уявіть футуристичне місто, де дані течуть як річки крізь квантові мережі. Код Нексус — це центральний вузол, де всі ці дані обробляються. Так само, як місто обробляє різні типи транспортних засобів (автомобілі, автобуси, велосипеди) на тих самих дорогах, Нексус використовує поліморфізм, щоб обробляти різні типи даних (числа, текст, журнали) через один і той же конвеєр. Кожен тип даних має свій власний "транспортний засіб" (обробник), але всі вони дотримуються тих самих "правил дорожнього руху" (інтерфейсу).

🔑 Ключові Концепції

1. Поліморфізм

Поліморфізм (від грецького poly = багато, morph = форма) дозволяє об'єктам різних класів оброблятися як об'єкти спільного базового класу. У Python це досягається за допомогою:

  • Спадкування: Підкласи успадковують від базового класу і перевизначають методи.
  • "Качина" Типізація: Якщо об'єкт "ходить як качка і крякає як качка", він вважається качкою — без явного спадкування.

Приклад: DataProcessor може перевіряти і обробляти дані, але конкретна поведінка залежить від того, чи це NumericProcessor, TextProcessor, чи LogProcessor.

2. Абстрактні Базові Класи (ABC)

ABC визначають контракт, якого повинні дотримуватися підкласи. Їх не можна інстанціювати безпосередньо, і вони можуть містити абстрактні методи (методи без реалізації), які підкласи повинні перевизначити.

Чому використовувати ABC?

  • Забезпечують консистентний інтерфейс для всіх підкласів.
  • Роблять код самодокументованим, чітко визначаючи очікувані методи.
  • Дозволяють перевірку на етапі виконання (наприклад, isinstance(obj, DataProcessor)).

3. "Качина" Типізація

"Качина" типізація — це концепція програмування, де тип або клас об'єкта визначається його поведінкою (методами і властивостями), а не його явним типом. У Python це часто підсумовується як:

"Якщо це ходить як качка і крякає як качка, то це, мабуть, качка."

Приклад: У Вправі 2 протокол ExportPlugin визначає, що будь-який клас з методом process_output може використовуватися як плагін, незалежно від його ієрархії спадкування.

4. Перевизначення Методів

Перевизначення методів дозволяє підкласу надати специфічну реалізацію методу, який уже визначений у його базовому класі. Саме так досягається поліморфізм в об'єктно-орієнтованому програмуванні.

Приклад: Метод validate у NumericProcessor перевіряє, чи є дані числовими, тоді як той самий метод у TextProcessor перевіряє, чи є дані рядком.

У цій вправі ви створите базову архітектуру для системи обробки даних. Метою є визначення спільного інтерфейсу для обробки різних типів даних (числа, текст, журнали), дозволяючи кожному типу зберігати свою унікальну поведінку.

🎯 Чому Абстрактні Класи?

Клас DataProcessor визначений як абстрактний базовий клас (ABC), тому що:

  • Його не можна інстанціювати безпосередньо — це шаблон для інших класів.
  • Він вимагає дотримання контракту: всі підкласи повинні реалізовувати validate і ingest.
  • Він надає єдиний інтерфейс для взаємодії з різними обробниками.

Реальна Аналогія: Уявіть DataProcessor як "USB-порт". Будь-який пристрій (підклас), який дотримується стандарту USB (реалізовує validate і ingest), може підключитися до нього.

🔧 Покрокова Реалізація

1. Визначте Абстрактний Базовий Клас

Почніть із створення класу DataProcessor з абстрактними методами. Використовуйте модуль abc, щоб визначити абстрактні методи.

from abc import ABC, abstractmethod from typing import Any class DataProcessor(ABC): # Абстрактний метод: повинен бути перевизначений у підкласах @abstractmethod def validate(self, data: Any) -> bool: """Перевірити, чи можуть дані бути оброблені цим обробником.""" pass # Абстрактний метод: повинен бути перевизначений у підкласах @abstractmethod def ingest(self, data: Any) -> None: """Обробити та зберегти дані всередині.""" pass # Конкретний метод: може бути успадкований або перевизначений def output(self) -> tuple[int, str]: """Вилучити найстаріші збережені дані та їх ранг.""" pass

💡 Краща Практика: Завжди використовуйте вказівки типів (наприклад, data: Any, -> bool), щоб зробити свій код самодокументованим. Це допомагає інструментам, таким як mypy, виявляти помилки на ранніх етапах.

2. Реалізуйте Спеціалізовані Обробники

Тепер створіть три підкласи DataProcessor: NumericProcessor, TextProcessor, і LogProcessor. Кожен з них перевизначить validate і ingest, щоб обробляти свій специфічний тип даних.

class NumericProcessor(DataProcessor): def __init__(self): self._data = [] # Внутрішнє сховище для оброблених даних self._rank = 0 # Лічильник рангу для виводу def validate(self, data: Any) -> bool: # Приймає int, float або списки цих типів if isinstance(data, (int, float)): return True if isinstance(data, list): return all(isinstance(item, (int, float)) for item in data) return False def ingest(self, data: int | float | list[int | float]) -> None: if not self.validate(data): raise ValueError("Некоректні числові дані") if isinstance(data, list): self._data.extend(map(str, data)) else: self._data.append(str(data)) def output(self) -> tuple[int, str]: if not self._data: raise IndexError("Немає даних для виводу") item = self._data.pop(0) rank = self._rank self._rank += 1 return (rank, item)
class TextProcessor(DataProcessor): def __init__(self): self._data = [] self._rank = 0 def validate(self, data: Any) -> bool: # Приймає str або списки str if isinstance(data, str): return True if isinstance(data, list): return all(isinstance(item, str) for item in data) return False def ingest(self, data: str | list[str]) -> None: if not self.validate(data): raise ValueError("Некоректні текстові дані") if isinstance(data, list): self._data.extend(data) else: self._data.append(data) def output(self) -> tuple[int, str]: if not self._data: raise IndexError("Немає даних для виводу") item = self._data.pop(0) rank = self._rank self._rank += 1 return (rank, item)
class LogProcessor(DataProcessor): def __init__(self): self._data = [] self._rank = 0 def validate(self, data: Any) -> bool: # Приймає dict зі строковими ключами/значеннями або список таких dict if isinstance(data, dict): return all(isinstance(k, str) and isinstance(v, str) for k, v in data.items()) if isinstance(data, list): return all(self.validate(item) for item in data) return False def ingest(self, data: dict[str, str] | list[dict[str, str]]) -> None: if not self.validate(data): raise ValueError("Некоректні дані журналу") if isinstance(data, list): for item in data: log_str = f"{item['log_level']}: {item['log_message']}" self._data.append(log_str) else: log_str = f"{data['log_level']}: {data['log_message']}" self._data.append(log_str) def output(self) -> tuple[int, str]: if not self._data: raise IndexError("Немає даних для виводу") item = self._data.pop(0) rank = self._rank self._rank += 1 return (rank, item)

⚠️ Поширені Помилки:

  • Забули викликати super().__init__(): Якщо ваш базовий клас має метод __init__, завжди викликайте його у підкласах, щоб забезпечити належну ініціалізацію.
  • Ігнорування вказівок типів: Хоча Python є динамічно типізованою мовою, вказівки типів покращують читабельність і допомагають виявляти помилки на ранніх етапах за допомогою інструментів, таких як mypy.
  • Не перевіряєте дані перед обробкою: Завжди перевіряйте дані у методі ingest, щоб уникнути прихованих помилок або псування даних.

Створіть екземпляри кожного обробника і протестуйте їхні методи. Зверніть особливу увагу на крайні випадки, такі як порожні списки або недійсні типи даних.

📝 Самооцінка

1. Чому DataProcessor визначений як абстрактний клас?

Відповідь: Щоб забезпечити дотримання контракту, згідно якого всі підкласи повинні реалізовувати validate і ingest. Це гарантує консистентність між усіма обробниками і запобігає інстанціюванню базового класу.

2. Що станеться, якщо викликати ingest без попередньої перевірки даних?

Відповідь: Метод викличе ValueError, якщо дані є недійсними, згідно з реалізацією методу ingest у кожному обробнику.

3. Як поліморфізм спрощує дизайн DataProcessor?

Відповідь: Поліморфізм дозволяє DataStream (у Вправі 1) обробляти всі обробники єдино через інтерфейс DataProcessor, без необхідності знати їхні конкретні реалізації.

🌊 Вправа 1: Поліморфна Обробка Потоку Даних

📚 Лекція: Динамічна Маршрутизація Даних

У Вправі 0 ви створили індивідуальні обробники. Тепер ви створите клас DataStream, який динамічно маршрутизує дані до відповідного обробника на основі поліморфізму. Саме тут проявляється потужність абстрактних класів і перевизначення методів: DataStream не потрібно знати деталі кожного обробника — він просто викликає validate і ingest на кожному зареєстрованому обробнику.

🎯 Чому Поліморфізм Важливий Тут

Поліморфізм дозволяє DataStream:

  • Обробляти всі обробники єдино: DataStream взаємодіє з обробниками через інтерфейс DataProcessor, незалежно від їхнього конкретного типу.
  • Додавати нові обробники без зміни коду: Ви можете зареєструвати новий обробник (наприклад, ImageProcessor) без зміни класу DataStream.
  • Обробляти різні типи даних: Потік може обробляти числа, текст, журнали або будь-які інші типи, для яких зареєстровано обробник.

Реальна Аналогія: Уявіть DataStream як поштову службу. Йому не потрібно знати вміст кожного пакета (даних) — він просто має знати, до якої поштової скриньки (обробника) його доставити на основі адреси (валідації).

🔧 Покрокова Реалізація

1. Визначте Клас DataStream

Клас DataStream буде:

  • Підтримувати список зареєстрованих обробників.
  • Маршрутизувати кожен елемент даних до першого обробника, який його валідує.
  • Реєструвати помилки для даних, які не можуть бути оброблені.
  • Відстежувати статистику для кожного обробника.
class DataStream: def __init__(self): self._processors = [] # Список зареєстрованих обробників def register_processor(self, proc: DataProcessor) -> None: """Зареєструвати новий обробник даних.""" self._processors.append(proc) def process_stream(self, stream: list[Any]) -> None: """Маршрутизувати кожен елемент у потоці до відповідного обробника.""" for data in stream: processed = False for proc in self._processors: if proc.validate(data): proc.ingest(data) processed = True break if not processed: print(f"Помилка DataStream - Не вдалося обробити елемент у потоці: {data}") def print_processors_stats(self) -> None: """Надрукувати статистику для всіх зареєстрованих обробників.""" for i, proc in enumerate(self._processors): processor_name = proc.__class__.__name__ data_count = len(proc._data) print(f"{processor_name}: всього {data_count} елементів оброблено, залишилося {data_count} в обробнику")

💡 Краща Практика: Метод process_stream використовує патерн цикл-і-вихід для маршрутизації даних до першого обробника, який їх валідує. Це ефективно і гарантує, що кожен елемент даних обробляється не більше ніж одним обробником.

2. Тестування DataStream

Зареєструйте всі три обробники і відправте змішаний потік даних через DataStream. Спостерігайте, як кожен елемент даних маршрутизується до правильного обробника.

# Ініціалізуємо DataStream stream = DataStream() # Реєструємо обробники stream.register_processor(NumericProcessor()) stream.register_processor(TextProcessor()) stream.register_processor(LogProcessor()) # Відправляємо пакет змішаних даних data = [ 42, "Hello", {'log_level': 'INFO', 'log_message': 'Test'}, [1, 2, 3], ["a", "b", "c"] ] stream.process_stream(data) # Друкуємо статистику stream.print_processors_stats()

Очікуваний Вивід:

Помилка DataStream - Не вдалося обробити елемент у потоці: Hello NumericProcessor: всього 2 елементів оброблено, залишилося 2 в обробнику TextProcessor: всього 1 елементів оброблено, залишилося 1 в обробнику LogProcessor: всього 1 елементів оброблено, залишилося 1 в обробнику

⚠️ Поширені Помилки:

  • Порядок реєстрації обробників має значення: Якщо два обробники можуть валідувати ті самі дані, перший зареєстрований обробник обробить їх. Переконайтеся, що обробники зареєстровані у правильному порядку.
  • Необроблювані дані ігноруються: Поточна реалізація друкує помилку, але продовжує обробку. Розгляньте можливість логування цих помилок для зручності наладкування.

🔍 Як Працює Поліморфізм Тут

Клас DataStream демонструє поліморфізм у дії:

  1. Єдиний Інтерфейс: DataStream викликає validate і ingest на кожному обробнику, не знаючи їхнього конкретного типу. Це можливо, тому що всі обробники успадковують від DataProcessor і реалізовують ці методи.
  2. Динамічна Диспетчеризація: Python автоматично викликає правильні методи validate і ingest на основі фактичного типу обробника (наприклад, NumericProcessor.validate vs. TextProcessor.validate).
  3. Розширюваність: Ви можете додати новий обробник (наприклад, ImageProcessor) без зміни класу DataStream. Просто зареєструйте його, і потік буде використовувати його автоматично.

📝 Самооцінка

1. Як DataStream визначає, який обробник використовувати для даного елемента даних?

Відповідь: Він ітерує через усі зареєстровані обробники і використовує перший, чий метод validate повертає True для даних.

2. Які переваги використання поліморфізму в цьому дизайні?

Відповідь:

  • Дозволяє DataStream обробляти нові типи даних без зміни коду.
  • Спрощує логіку DataStream, обробляючи всі обробники єдино.
  • Робить систему більш підтримуваною і розширюваною.

3. Як ви розширите цю систему для підтримки нового типу даних (наприклад, ImageProcessor)?

Відповідь: Створіть новий клас (наприклад, ImageProcessor), який успадковує від DataProcessor і реалізовує validate і ingest. Потім зареєструйте його в DataStream.

У Вправі 1 ви побудували систему, яка обробляє і зберігає дані. Тепер ви розширите її, щоб експортувати оброблені дані за допомогою системи плагінів. 🎯 Чому "Качина" Типізація?

"Качина" типізація дозволяє:

  • Уникнути тісної зв'язності: Плагінам не потрібно успадковуватися від конкретного класу — їм просто потрібно реалізовувати необхідні методи (наприклад, process_output).
  • Підтримувати кілька реалізацій: Ви можете створити плагіни для CSV, JSON або XML без зміни класу DataStream.
  • Зберегти гнучкість коду: Нові формати експорту можуть бути додані без зміни існуючого коду.

Реальна Аналогія: Уявіть плагіни як "адаптери" для електричної розетки. Якщо адаптер (плагін) має правильні штекери (методи), він може підключитися до розетки (DataStream), незалежно від його марки або дизайну.

🔍 Протоколи в Python

Протокол typing.Protocol у Python дозволяє визначити "контракт" для класів без вимоги успадкування від конкретного базового класу. Це ідеально підходить для систем плагінів, де ви хочете забезпечити наявність певних методів без обмеження спадкування.

Приклад: Протокол ExportPlugin вимагає наявності методу process_output. Будь-який клас, який реалізовує цей метод, може використовуватися як плагін, навіть якщо він не успадковується від конкретного класу.

🔧 Покрокова Реалізація

1. Визначте Протокол ExportPlugin

Використовуйте typing.Protocol, щоб визначити інтерфейс для плагінів експорту. Декоратор @runtime_checkable дозволяє використовувати isinstance для перевірки, чи відповідає об'єкт протоколу.

from typing import Protocol, runtime_checkable @runtime_checkable class ExportPlugin(Protocol): def process_output(self, data: list[tuple[int, str]]) -> None: """Обробити та експортувати дані.""" pass

2. Реалізуйте Плагіни CSV і JSON

Створіть два плагіни: CSVPlugin і JSONPlugin. Кожен з них реалізує process_output, щоб форматувати дані як CSV або JSON відповідно.

class CSVPlugin: def process_output(self, data: list[tuple[int, str]]) -> None: """Експортувати дані у форматі CSV.""" csv_data = [f"{rank},"{value}" for rank, value in data] print("Вивід CSV:\n" + "\n".join(csv_data)) class JSONPlugin: def process_output(self, data: list[tuple[int, str]]) -> None: """Експортувати дані у форматі JSON.""" json_data = {f"item_{rank}": value for rank, value in data} print("Вивід JSON:\n" + str(json_data))

3. Розширте DataStream для Підтримки Плагінів

Додайте метод output_pipeline до DataStream, який споживає nb елементів з усіх зареєстрованих обробників і експортує їх за допомогою наданого плагіна.

class DataStream: # ... (попередні методи) def output_pipeline(self, nb: int, plugin: ExportPlugin) -> None: """Спожити `nb` елементів з усіх обробників і експортувати за допомогою плагіна.""" all_data = [] for proc in self._processors: for _ in range(nb): try: all_data.append(proc.output()) except IndexError: break plugin.process_output(all_data)

4. Тестування Конвеєра

Зареєструйте обробники, обробіть потік даних, а потім експортуйте результати за допомогою плагінів CSV і JSON.

# Ініціалізуємо DataStream stream = DataStream() # Реєструємо обробники stream.register_processor(NumericProcessor()) stream.register_processor(TextProcessor()) stream.register_processor(LogProcessor()) # Обробляємо потік даних data = [ 42, "Hello", {'log_level': 'INFO', 'log_message': 'Test'}, [1, 2, 3], ["a", "b", "c"] ] stream.process_stream(data) # Експортуємо 2 елементи з кожного обробника у форматі CSV stream.output_pipeline(2, CSVPlugin()) # Експортуємо 2 елементи з кожного обробника у форматі JSON stream.output_pipeline(2, JSONPlugin())

Очікуваний Вивід:

Вивід CSV: 0,42 1,1 0,Hello 0,INFO: Test Вивід JSON: {'item_0': '42', 'item_1': '1', 'item_0': 'Hello', 'item_0': 'INFO: Test'}

💡 Краща Практика: При проектуванні систем плагінів:

  • Використовуйте протоколи, щоб визначити інтерфейси без вимоги спадкування.
  • Тримайте методи плагінів простими і зосередженими (наприклад, process_output обробляє лише експорт).
  • Документуйте очікуваний вхід/вихід для методів плагінів (наприклад, data: list[tuple[int, str]]).

⚠️ Поширені Помилки:

  • Припущення, що плагіни не мають стану: Якщо ваші плагіни повинні зберігати стан (наприклад, дескриптори файлів), переконайтеся, що вони спроектовані для безпечної роботи з ним.
  • Ігнорування вказівок типів у протоколах: Завжди використовуйте вказівки типів у протоколах, щоб зробити очікування ясними.
  • Необробка порожніх даних: Метод output_pipeline повинен коректно обробляти випадки, коли обробники мають менше ніж nb елементів.

🔍 Як "Качина" Типізація Забезпечує Гнучкість

Система плагінів у цій вправі демонструє потужність "качиної" типізації:

  1. Не Потрібно Спадкування: CSVPlugin і JSONPlugin не успадковують від спільного базового класу. Їм просто потрібно реалізовувати process_output.
  2. Гнучкість на Етапі Виконання: Ви можете передати будь-який об'єкт, який реалізовує process_output до output_pipeline, навіть якщо він визначений пізніше (наприклад, в іншому модулі).
  3. Легке Тестування: Ви можете створювати мок-плагіни для тестування без зміни класу DataStream.

📝 Самооцінка

1. Що таке "качина" типізація, і чим вона відрізняється від поліморфізму на основі спадкування?

Відповідь: "Качина" типізація визначає придатність об'єкта на основі його методів/властивостей, а не його типу. Поліморфізм на основі спадкування вимагає, щоб об'єкти успадковувалися від конкретного класу. "Качина" типізація є більш гнучкою.

2. Чому використовувати typing.Protocol для плагінів?

Відповідь: Щоб визначити контракт (інтерфейс), якого повинні дотримуватися плагіни, без вимоги успадкування від конкретного класу. Це дозволяє використовувати "качину" типізацію, забезпечуючи при цьому типову безпеку.

3. Як ви додасте новий формат експорту (наприклад, XML)?

Відповідь: Створіть новий клас (наприклад, XMLPlugin), який реалізовує метод process_output. Зміни до DataStream не потрібні.

🎓 Висновок: Ключові Висновки

У цьому модулі ви дізналися, як використовувати поліморфізм, абстрактні класи і "качину" типізацію для побудови гнучкої та розширюваної системи обробки даних. Ось короткий огляд ключових концепцій:

🔑 Основні Концепції

  1. Поліморфізм: Обробляйте об'єкти різних класів єдино через спільний інтерфейс. Це дозволяє створювати гнучкий і багаторазово використовуваний код.
  2. Абстрактні Базові Класи (ABC): Визначте контракти для підкласів, щоб забезпечити консистентність. Використовуйте abc.ABC і @abstractmethod для цього.
  3. Перевизначення Методів: Налаштуйте поведінку в підкласах, перевизначаючи методи базового класу.
  4. "Качина" Типізація: Зосереджуйтеся на поведінці об'єкта (методах/властивостях), а не на його типі. Дозволяє створювати гнучкі системи плагінів.
  5. Протоколи: Визначте інтерфейси без спадкування. Використовуйте typing.Protocol для типової безпеки.

🏗️ Принципи Проектування

  • Розділення Обов'язків: Кожен клас (наприклад, NumericProcessor, CSVPlugin) має єдину відповідальність.
  • Принцип Відкритості/Закритості: Клас DataStream відкритий для розширення (нові обробники/плагіни), але закритий для зміни.
  • Слабка Зв'язність: DataStream залежить від абстракцій (DataProcessor, ExportPlugin), а не від конкретних реалізацій.
