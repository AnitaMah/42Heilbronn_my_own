🔑 Key Concepts

1. Polymorphism

Polymorphism (from Greek poly = many, morph = form) allows objects of different classes to be treated as objects of a common superclass. In Python, this is achieved through:

  • Inheritance: Subclasses inherit from a base class and override methods.
  • Duck Typing: If an object "walks like a duck and quacks like a duck," it’s treated as a duck—no explicit inheritance required.

Example: A DataProcessor can validate and ingest data, but the specific behavior depends on whether it's a NumericProcessor, TextProcessor, or LogProcessor.

2. Abstract Base Classes (ABCs)

ABCs define a contract that subclasses must follow. They cannot be instantiated directly and may contain abstract methods (methods without implementation) that subclasses must override.

Why use ABCs?

  • Enforce a consistent interface across subclasses.
  • Make your code self-documenting by clearly defining expected methods.
  • Enable runtime checks (e.g., isinstance(obj, DataProcessor)).

3. Duck Typing

Duck typing is a programming concept where the type or class of an object is determined by its behavior (methods and properties) rather than its explicit type. In Python, this is often summarized as:

"If it walks like a duck and quacks like a duck, then it must be a duck."

Example: In Exercise 2, the ExportPlugin protocol defines that any class with a process_output method can be used as a plugin, regardless of its inheritance hierarchy.

4. Method Overriding

Method overriding allows a subclass to provide a specific implementation of a method that is already defined in its superclass. This is how polymorphism is achieved in object-oriented programming.

Example: The validate method in NumericProcessor checks if the data is numeric, while the same method in TextProcessor checks if the data is a string.

🛠️ Exercise 0: Data Processor

📚 Lecture: Building the Foundation

In this exercise, you’ll create the base architecture for a data processing system. The goal is to define a common interface for processing different types of data (numeric, text, logs) while allowing each type to maintain its unique behavior.

🎯 Why Abstract Classes?

The DataProcessor class is defined as an abstract base class (ABC) because:

  • It cannot be instantiated directly—it’s a blueprint for other classes.
  • It enforces a contract: All subclasses must implement validate and ingest.
  • It provides a unified interface for interacting with different processors.

Real-World Analogy: Think of DataProcessor as a "USB port." Any device (subclass) that follows the USB standard (implements validate and ingest) can plug into it.

🔧 Step-by-Step Implementation

1. Define the Abstract Base Class

Start by creating the DataProcessor class with abstract methods. Use the abc module to define abstract methods.

from abc import ABC, abstractmethod from typing import Any class DataProcessor(ABC): # Abstract method: Must be overridden by subclasses @abstractmethod def validate(self, data: Any) -> bool: """Check if the data can be processed by this processor.""" pass # Abstract method: Must be overridden by subclasses @abstractmethod def ingest(self, data: Any) -> None: """Process and store the data internally.""" pass # Concrete method: Can be inherited or overridden def output(self) -> tuple[int, str]: """Extract the oldest stored data and its rank.""" pass

💡 Best Practice: Always use type hints (e.g., data: Any, -> bool) to make your code self-documenting. This helps tools like mypy catch type-related errors early.

2. Implement Specialized Processors

Now, create three subclasses of DataProcessor: NumericProcessor, TextProcessor, and LogProcessor. Each will override validate and ingest to handle its specific data type.

class NumericProcessor(DataProcessor): def __init__(self): self._data = [] # Internal storage for processed data self._rank = 0 # Rank counter for output def validate(self, data: Any) -> bool: # Accept int, float, or lists of these types return isinstance(data, (int, float, list)) def ingest(self, data: int | float | list[int | float]) -> None: if not self.validate(data): raise ValueError("Improper numeric data") if isinstance(data, list): self._data.extend(map(str, data)) else: self._data.append(str(data)) def output(self) -> tuple[int, str]: if not self._data: raise IndexError("No data to output") item = self._data.pop(0) rank = self._rank self._rank += 1 return (rank, item)
class TextProcessor(DataProcessor): def __init__(self): self._data = [] self._rank = 0 def validate(self, data: Any) -> bool: return isinstance(data, (str, list)) def ingest(self, data: str | list[str]) -> None: if not self.validate(data): raise ValueError("Improper text data") if isinstance(data, list): self._data.extend(data) else: self._data.append(data) def output(self) -> tuple[int, str]: if not self._data: raise IndexError("No data to output") item = self._data.pop(0) rank = self._rank self._rank += 1 return (rank, item)
class LogProcessor(DataProcessor): def __init__(self): self._data = [] self._rank = 0 def validate(self, data: Any) -> bool: # Accept dict with string key-value pairs or list of such dicts if isinstance(data, dict): return all(isinstance(k, str) and isinstance(v, str) for k, v in data.items()) elif isinstance(data, list): return all(self.validate(item) for item in data) return False def ingest(self, data: dict[str, str] | list[dict[str, str]]) -> None: if not self.validate(data): raise ValueError("Improper log data") if isinstance(data, list): for item in data: log_str = f"{item['log_level']}: {item['log_message']}" self._data.append(log_str) else: log_str = f"{data['log_level']}: {data['log_message']}" self._data.append(log_str) def output(self) -> tuple[int, str]: if not self._data: raise IndexError("No data to output") item = self._data.pop(0) rank = self._rank self._rank += 1 return (rank, item)

⚠️ Common Pitfalls:

  • Forgetting to call super().__init__(): If your base class has an __init__ method, always call it in subclasses to ensure proper initialization.
  • Ignoring type hints: While Python is dynamically typed, type hints improve readability and help catch errors early with tools like mypy.
  • Not validating data before ingestion: Always validate data in ingest to avoid silent failures or corruption.

3. Test Your Implementation

Create instances of each processor and test their methods. Pay special attention to edge cases, such as empty lists or invalid data types.

# Test NumericProcessor numeric_proc = NumericProcessor() print(numeric_proc.validate(42)) # True print(numeric_proc.validate("Hello")) # False numeric_proc.ingest([1, 2, 3]) print(numeric_proc.output()) # (0, '1') print(numeric_proc.output()) # (1, '2') # Test invalid ingestion (should raise ValueError) try: numeric_proc.ingest("foo") except ValueError as e: print(f"Got exception: {e}") # Got exception: Improper numeric data

🌊 Exercise 1: Polymorphic Processing of a Data Stream

📚 Lecture: Routing Data Dynamically

In Exercise 0, you built individual processors. Now, you’ll create a DataStream class that dynamically routes data to the appropriate processor based on polymorphism. This is where the power of abstract classes and method overriding shines: the DataStream doesn’t need to know the specifics of each processor—it just calls validate and ingest on each registered processor.

🎯 Why Polymorphism Matters Here

Polymorphism allows the DataStream to:

  • Treat all processors uniformly: The DataStream interacts with processors through the DataProcessor interface, regardless of their specific type.
  • Add new processors without modification: You can register a new processor (e.g., ImageProcessor) without changing the DataStream class.
  • Handle diverse data types: The stream can process numbers, text, logs, or any other type for which a processor is registered.

Real-World Analogy: Think of the DataStream as a postal service. It doesn’t need to know the contents of each package (data)—it just needs to know which mailbox (processor) to deliver it to based on the address (validation).

🔧 Step-by-Step Implementation

1. Define the DataStream Class

The DataStream class will:

  • Maintain a list of registered processors.
  • Route each data element to the first processor that validates it.
  • Log errors for unprocessable data.
  • Track statistics for each processor.
class DataStream: def __init__(self): self._processors = [] # List of registered processors def register_processor(self, proc: DataProcessor) -> None: """Register a new data processor.""" self._processors.append(proc) def process_stream(self, stream: list[Any]) -> None: """Route each element in the stream to the appropriate processor.""" for data in stream: processed = False for proc in self._processors: if proc.validate(data): proc.ingest(data) processed = True break if not processed: print(f"DataStream error - Can't process element in stream: {data}") def print_processors_stats(self) -> None: """Print statistics for all registered processors.""" for i, proc in enumerate(self._processors): processor_name = proc.__class__.__name__ data_count = len(proc._data) print(f"{processor_name}: total {data_count} items processed, remaining {data_count} on processor")

💡 Best Practice: The process_stream method uses a loop-and-break pattern to route data to the first processor that validates it. This is efficient and ensures that each data element is processed by at most one processor.

2. Test the DataStream

Register all three processors and send a mixed data stream through the DataStream. Observe how each data element is routed to the correct processor.

# Initialize DataStream stream = DataStream() # Register processors stream.register_processor(NumericProcessor()) stream.register_processor(TextProcessor()) stream.register_processor(LogProcessor()) # Send a batch of mixed data data = [ 42, "Hello", {'log_level': 'INFO', 'log_message': 'Test'}, [1, 2, 3], ["a", "b", "c"] ] stream.process_stream(data) # Print statistics stream.print_processors_stats()

Expected Output:

DataStream error - Can't process element in stream: Hello NumericProcessor: total 2 items processed, remaining 2 on processor TextProcessor: total 1 items processed, remaining 1 on processor LogProcessor: total 1 items processed, remaining 1 on processor

⚠️ Common Pitfalls:

  • Order of processor registration matters: If two processors can validate the same data, the first one registered will handle it. Ensure processors are registered in the correct order.
  • Unprocessable data is silently ignored: The current implementation prints an error but continues processing. Consider logging these errors for debugging.

🔍 How Polymorphism Works Here

The DataStream class demonstrates polymorphism in action:

  1. Uniform Interface: The DataStream calls validate and ingest on each processor without knowing its specific type. This is possible because all processors inherit from DataProcessor and implement these methods.
  2. Dynamic Dispatch: Python automatically calls the correct validate and ingest methods based on the actual type of the processor (e.g., NumericProcessor.validate vs. TextProcessor.validate).
  3. Extensibility: You can add a new processor (e.g., ImageProcessor) without modifying the DataStream class. Just register it, and the stream will use it automatically.

📝 Self-Assessment Quiz

1. How does DataStream determine which processor to use for a given data element?

Answer: It iterates through all registered processors and uses the first one whose validate method returns True for the data.

2. What are the benefits of using polymorphism in this design?

Answer:

  • Allows DataStream to handle new data types without modification.
  • Simplifies the DataStream logic by treating all processors uniformly.
  • Makes the system more maintainable and extensible.

3. How would you extend this system to support a new data type (e.g., ImageProcessor)?

Answer: Create a new class (e.g., ImageProcessor) that inherits from DataProcessor and implements validate and ingest. Then register it with the DataStream.

🚀 Exercise 2: Data Pipeline

📚 Lecture: Duck Typing and Plugin Systems

In Exercise 1, you built a system that processes and stores data. Now, you’ll extend it to export the processed data using a plugin system. This is where duck typing and protocols come into play. Instead of requiring plugins to inherit from a specific class, you’ll define a protocol (a set of methods) that plugins must implement.

🎯 Why Duck Typing?

Duck typing allows you to:

  • Avoid tight coupling: Plugins don’t need to inherit from a specific class—they just need to implement the required methods (e.g., process_output).
  • Support multiple implementations: You can create CSV, JSON, or XML plugins without modifying the DataStream class.
  • Keep the code flexible: New export formats can be added without changing existing code.

Real-World Analogy: Think of plugins as "adapters" for a power outlet. As long as the adapter (plugin) has the correct prongs (methods), it can plug into the outlet (DataStream), regardless of its brand or design.

🔍 Protocols in Python

Python’s typing.Protocol allows you to define a "contract" for classes without requiring them to inherit from a specific base class. This is perfect for plugin systems where you want to enforce a set of methods without restricting inheritance.

Example: The ExportPlugin protocol requires a process_output method. Any class that implements this method can be used as a plugin, even if it doesn’t inherit from a specific class.

🔧 Step-by-Step Implementation

1. Define the ExportPlugin Protocol

Use typing.Protocol to define the interface for export plugins. The @runtime_checkable decorator allows you to use isinstance to check if an object conforms to the protocol.

from typing import Protocol, runtime_checkable @runtime_checkable class ExportPlugin(Protocol): def process_output(self, data: list[tuple[int, str]]) -> None: """Process and export the data.""" pass

2. Implement CSV and JSON Plugins

Create two plugins: CSVPlugin and JSONPlugin. Each will implement process_output to format the data as CSV or JSON, respectively.

class CSVPlugin: def process_output(self, data: list[tuple[int, str]]) -> None: """Export data as CSV.""" csv_data = [f"{rank},"{value}" for rank, value in data] print("CSV Output:\n" + "\n".join(csv_data)) class JSONPlugin: def process_output(self, data: list[tuple[int, str]]) -> None: """Export data as JSON.""" json_data = {f"item_{rank}": value for rank, value in data} print("JSON Output:\n" + str(json_data))

3. Extend DataStream to Support Plugins

Add an output_pipeline method to DataStream that consumes nb elements from all registered processors and exports them using the provided plugin.

class DataStream: # ... (previous methods) def output_pipeline(self, nb: int, plugin: ExportPlugin) -> None: """Consume `nb` elements from all processors and export using the plugin.""" all_data = [] for proc in self._processors: for _ in range(nb): try: all_data.append(proc.output()) except IndexError: break plugin.process_output(all_data)

4. Test the Pipeline

Register processors, process a stream of data, and then export the results using the CSV and JSON plugins.

# Initialize DataStream stream = DataStream() # Register processors stream.register_processor(NumericProcessor()) stream.register_processor(TextProcessor()) stream.register_processor(LogProcessor()) # Process a stream of data data = [ 42, "Hello", {'log_level': 'INFO', 'log_message': 'Test'}, [1, 2, 3], ["a", "b", "c"] ] stream.process_stream(data) # Export 2 elements from each processor as CSV stream.output_pipeline(2, CSVPlugin()) # Export 2 elements from each processor as JSON stream.output_pipeline(2, JSONPlugin())

Expected Output:

CSV Output: 0,42 1,1 0,Hello 0,INFO: Test JSON Output: {'item_0': '42', 'item_1': '1', 'item_0': 'Hello', 'item_0': 'INFO: Test'}

💡 Best Practice: When designing plugin systems:

  • Use protocols to define interfaces without requiring inheritance.
  • Keep plugin methods simple and focused (e.g., process_output only handles exporting).
  • Document the expected input/output for plugin methods (e.g., data: list[tuple[int, str]]).

⚠️ Common Pitfalls:

  • Assuming plugins are stateless: If your plugins need to maintain state (e.g., file handles), ensure they are designed to handle it safely.
  • Ignoring type hints in protocols: Always use type hints in protocols to make expectations clear.
  • Not handling empty data: The output_pipeline method should gracefully handle cases where processors have fewer than nb elements.

🔍 How Duck Typing Enables Flexibility

The plugin system in this exercise demonstrates the power of duck typing:

  1. No Inheritance Required: CSVPlugin and JSONPlugin don’t inherit from a common base class. They only need to implement process_output.
  2. Runtime Flexibility: You can pass any object that implements process_output to output_pipeline, even if it’s defined later (e.g., in another module).
  3. Easy Testing: You can create mock plugins for testing without modifying the DataStream class.

📝 Self-Assessment Quiz

1. What is duck typing, and how does it differ from inheritance-based polymorphism?

Answer: Duck typing determines an object's suitability based on its methods/properties, not its type. Inheritance-based polymorphism requires objects to inherit from a specific class. Duck typing is more flexible.

2. Why use typing.Protocol for plugins?

Answer: To define a contract (interface) that plugins must follow without requiring them to inherit from a specific class. This enables duck typing while providing type safety.

3. How would you add a new export format (e.g., XML)?

Answer: Create a new class (e.g., XMLPlugin) that implements the process_output method. No changes to DataStream are needed.

🎓 Conclusion: Key Takeaways

In this module, you’ve learned how to use polymorphism, abstract classes, and duck typing to build a flexible and extensible data processing pipeline. Here’s a recap of the key concepts:

🔑 Core Concepts

  1. Polymorphism: Treat objects of different classes uniformly through a common interface. This enables flexible and reusable code.
  2. Abstract Base Classes (ABCs): Define contracts for subclasses to ensure consistency. Use abc.ABC and @abstractmethod to enforce this.
  3. Method Overriding: Customize behavior in subclasses by overriding methods from the base class.
  4. Duck Typing: Focus on an object’s behavior (methods/properties) rather than its type. Enables flexible plugin systems.
  5. Protocols: Define interfaces without inheritance. Use typing.Protocol for type safety.

🏗️ Design Principles

  • Separation of Concerns: Each class (e.g., NumericProcessor, CSVPlugin) has a single responsibility.
  • Open/Closed Principle: The DataStream class is open for extension (new processors/plugins) but closed for modification.
  • Loose Coupling: DataStream depends on abstractions (DataProcessor, ExportPlugin), not concrete implementations.
